# Implementation Plan

## 1. 文档目标

本文档将前述 10 份方案细化为可执行的落地任务计划，面向 AI 开发者逐步实施。计划遵循严格开发流程，先做基础闭环，再接入语音、状态机、数字人、多模态、知识库、评测与部署。每一步都必须单独完成、单独验证，未通过验证不得进入下一步。

## 2. 执行总则

- 一次只交给 AI 开发者一个步骤，不允许跨步混做。
- 每一步结束后必须提交产出说明、测试结果和遗留问题。
- 每一步必须能在当前主干上独立运行，不允许“先写一半等后面再补”。
- 先保证文本主链路稳定，再接语音，再接数字人，再接多模态和知识库。
- 任何模型服务都必须先有 mock 或 demo 路径，保证联调和录屏随时可进行。
- 每日结束前必须执行一次当前阶段的回归验证，确保新增功能没有破坏已有链路。
- `data/val` 中的企业数据视为只读验证集，原始文件不得直接修改，所有清洗结果、索引文件和中间产物都必须输出到独立衍生目录。
- 任何读取 `data/val` 的流程都必须显式过滤 `._*` 和 `.DS_Store` 一类无效文件，避免污染样本计数和评测结果。

## 3. 第一阶段：工程骨架与基础设施

### 步骤 1：建立仓库目录骨架

- 指令：只创建基础目录结构，至少包含前端、网关、编排、各服务、共享库、数据、基础设施、文档、测试目录。不要加入业务逻辑。
- 产出：清晰的目录树和总 README 的目录说明。
- 验证测试：检查目录树是否完整，目录命名是否与总方案一致，README 是否写明每个目录职责。
- 通过标准：任何新成员打开仓库后，能仅通过目录和 README 理解系统分层。

### 步骤 2：建立环境变量清单

- 指令：新增环境配置说明文件，列出所有基础服务和模型服务需要的变量名、用途、默认值说明和是否必填。
- 产出：统一环境变量清单和样例配置文件。
- 验证测试：逐项检查网关、编排、数据库、缓存、对象存储、LLM、ASR、TTS、数字人服务是否都有对应配置项。
- 通过标准：不存在“代码里硬编码但配置清单里缺失”的关键变量。

### 步骤 3：建立共享契约文档

- 指令：只定义跨服务共享的数据契约，包括会话信息、事件格式、转写结果、对话结果、数字人驱动参数和错误响应格式。
- 产出：共享字段说明文档。
- 验证测试：检查所有关键流程是否都有明确输入输出字段，字段命名是否统一，是否存在含义重复或冲突的字段。
- 通过标准：后续服务开发不需要再重新猜测 JSON 结构。

### 步骤 4：启动基础设施容器

- 指令：仅启动数据库、缓存和对象存储三类基础服务，并为它们配置健康检查和持久化卷。
- 产出：基础设施启动配置和运行说明。
- 验证测试：启动后检查三类服务是否均处于健康状态；重启后检查持久化数据是否仍存在。
- 通过标准：基础设施可稳定重复启动，不依赖手工修复。

### 步骤 5：建立数据库初始表结构

- 指令：只创建会话、消息、系统事件、评测记录和媒体索引的初始表结构，不接业务接口。
- 产出：数据库初始化脚本和表结构说明。
- 验证测试：人工插入一条会话和一条消息记录，确认主键、外键、时间字段和状态字段可正常保存。
- 通过标准：表结构足以支撑文本闭环和后续日志扩展。

### 步骤 6：准备 Demo 数据目录

- 指令：建立演示数据目录，至少放入一段文本对话脚本、一段样例音频说明、一段样例视频帧说明和导出日志说明。
- 产出：可复用的 Demo 数据清单。
- 验证测试：检查 Demo 资源是否能被 README 明确引用，是否能支撑后续 mock 联调。
- 通过标准：即使真实模型未接入，也能基于 Demo 数据演示主流程。

### 步骤 6A：审计企业验证集结构

- 指令：只读取 `data/val` 的目录结构和样本组织方式，梳理 NoXI 与 RECOLA 两套数据的模态目录、会话目录、角色目录和片段编号规则，不进行任何模型训练。
- 产出：企业验证集结构说明文档。
- 验证测试：确认至少识别出四类资源目录：音频、视频、情绪标签、3D 面部特征；确认 NoXI 含双角色目录，RECOLA 存在视频/音频角色编号与情绪标签编号不一致的问题。
- 通过标准：团队对企业验证集的目录结构、命名方式和潜在陷阱达成统一认知。

### 步骤 6B：建立跨模态 manifest

- 指令：按“数据集、会话、角色、片段编号”建立 manifest，把同一片段的音频、视频、情绪 CSV 和 3D 特征文件映射到同一条记录上。
- 产出：跨模态样本清单。
- 验证测试：随机抽查 NoXI 和 RECOLA 各一组样本，确认 manifest 能正确定位对应的音频、视频、情绪标签和 3D 特征；确认不会把 `P1/P2` 与 `P41/P42` 之类不同命名体系混错。
- 通过标准：后续所有训练、验证和回放流程都能只依赖 manifest，而不是现场拼路径。

### 步骤 6C：建立标签映射规范

- 指令：将企业数据中的 `AU`、`valence`、`arousal`、离散情绪概率字段统一映射到项目内部标签空间，包括基础情绪状态、置信度和可解释证据字段。
- 产出：标签映射说明文档。
- 验证测试：任选两段高正向和高负向样本，确认通过映射后能得到项目可消费的统一标签，不会出现“源标签存在但项目标签为空”的情况。
- 通过标准：多模态模块后续读取企业标签时不再临时写规则。

### 步骤 6D：建立数据预处理规范

- 指令：只定义企业数据接入前的预处理要求，包括音频重采样、声道规范、视频抽帧规范、CSV 编码规范、`_full.npy` 与普通 `.npy` 的使用边界。
- 产出：数据预处理规范文档。
- 验证测试：检查规范中是否明确说明企业音频是 `44.1kHz` 立体声，需要转换为 ASR 侧使用的单声道标准；检查是否说明 3D 特征与情绪 CSV 的对齐关系。
- 通过标准：数据接入前不会因采样率、声道或文件类型不一致导致后续模块反复返工。

### 步骤 6E：建立文本通道验证来源

- 指令：针对企业验证集中缺少现成文本转录的问题，单独制定文本通道验证方案，明确哪些样本采用人工转录，哪些样本允许先由 ASR 生成再人工校对。
- 产出：文本验证样本来源规范。
- 验证测试：抽取一小批音频样本，确认每条样本最终都能落到“有可用文本”的状态，并记录文本来源是人工还是机器初稿。
- 通过标准：文本情绪分析和对话回放不再被“没有现成转录”卡住。

### 步骤 6F：扩展完整验证集 manifest

- 指令：把样例级 manifest 扩展为完整验证集清单，要求覆盖当前 `data/val` 下全部可用 NoXI 和 RECOLA 样本，并冻结一个可复现的 manifest 版本。
- 产出：完整 `data/manifests/val_manifest.jsonl`。
- 验证测试：检查 manifest 记录总数是否稳定，当前基线应为 `1126`；其中 NoXI `1106`、RECOLA `20`；随机抽查多条记录确认路径存在。
- 通过标准：后续数据相关开发都能围绕同一份完整 manifest 进行，而不是围绕少量样例或手工路径。

### 步骤 6G：生成全量转录模板

- 指令：依据完整 manifest 生成一份一一对应的转录模板文件，模板中必须预置 `workflow_status`、`next_action`、`asr_draft_status`、`review_status`、`review_decision`、`final_text` 等工作流字段，用于后续 ASR 初稿回填和人工校对。
- 产出：`data/derived/transcripts/val_transcripts_template.jsonl`。
- 验证测试：检查模板记录数是否与 manifest 完全一致，当前基线应同为 `1126`；任意一条模板记录都能通过 `record_id` 回查原始音频；任意抽查一条记录都包含完整的转录工作流字段。
- 通过标准：文本链路补齐工作从此有统一入口和统一状态机，不再靠零散笔记或临时表格管理。

### 步骤 6H：生成 QC 报告与数据门槛

- 指令：基于完整 manifest 和转录模板生成首版数据质检报告，并把会阻断后续模型工作的质量问题明确写成门槛；报告必须按数据集和规范角色拆表。
- 产出：`data/derived/qc_report.md`。
- 验证测试：报告必须包含当前基线数字：完整多模态记录 `1124`、缺失情绪标签 `2`、隐藏文件过滤数 `2486`、对齐错位 `1124`；必须包含 `Coverage By Dataset And Role`、`Transcript Workflow By Dataset And Role`、问题样本分表和后续人工处理事项。
- 通过标准：在数据质量问题和转录补齐进度被显式量化之前，不进入多模态训练和正式评测阶段。

### 步骤 6I：固化转录状态机与回填规则

- 指令：只定义转录工作流状态推进规则，至少覆盖 `pending_asr`、`draft_ready`、`pending_review`、`verified` 四种主状态，并明确哪些字段由 ASR 批处理写入、哪些字段只能由人工复核写入、哪些字段一旦锁定不得回退。
- 产出：转录工作流说明和字段写入边界。
- 验证测试：任选三条模板记录，分别模拟“刚生成初稿”“进入人工复核”“人工确认完成”三种场景，检查状态、文本字段和审核字段变化是否符合规则。
- 通过标准：后续 ASR 初稿生成、人工复核和正式评测都能围绕同一状态机执行，不会出现字段被错误覆盖。

## 4. 第二阶段：文本主链路最小闭环

### 步骤 7：搭建前端单页骨架

- 指令：只实现单页控制台布局，包含采集区、数字人区、转写区、情绪区、聊天流、会话控制区六个面板，先全部使用静态占位内容。
- 产出：可打开的前端页面骨架。
- 验证测试：页面首次加载无白屏、无明显布局错位、无控制台报错，六个面板都可见。
- 通过标准：页面结构稳定，可作为后续所有功能挂载容器。

### 步骤 8：实现会话创建接口

- 指令：只实现“创建会话”能力，由网关生成会话标识并写入数据库，不处理任何消息。
- 产出：会话创建接口和数据库写入逻辑。
- 验证测试：从页面点击“开始会话”后，页面能得到新的会话标识，数据库中能查到对应会话记录。
- 通过标准：会话创建过程稳定且幂等。

### 步骤 9：前端接入会话启动流程

- 指令：将前端开始按钮接入会话创建接口，页面显示当前会话编号和基础状态，不接入实时消息。
- 产出：可操作的会话启动流程。
- 验证测试：刷新页面后重新创建会话，编号变化符合预期；旧会话不会污染新会话状态。
- 通过标准：用户能明确知道自己当前处于哪个会话。

### 步骤 10：建立前后端实时连接

- 指令：仅实现会话级实时连接、心跳和断线重连机制，不推送业务消息。
- 产出：可建立和恢复的实时连接。
- 验证测试：建立连接后主动断网再恢复，确认页面能够重连且不会卡死；日志中能看到连接状态变化。
- 通过标准：实时连接具备基本稳定性。

### 步骤 11：实现文本输入提交流程

- 指令：只支持用户输入文本并提交到网关，网关将消息写入数据库并回传确认事件，不接入编排服务。
- 产出：文本消息提交链路。
- 验证测试：提交后页面能看到“发送成功”状态，数据库中保存用户消息内容、时间戳和会话标识。
- 通过标准：文本输入可稳定入库且不丢消息。

### 步骤 12：实现编排服务的 mock 回复

- 指令：只做最简单的 mock 编排流程，收到文本后返回固定结构的回复结果，字段必须符合共享契约。
- 产出：mock 回复事件和基础状态切换。
- 验证测试：提交不同文本后，页面都能收到结构合法的回复事件，缺失字段或非法阶段会被拦截。
- 通过标准：即使没有真实模型，文本问答闭环已经跑通。

### 步骤 13：前端展示聊天流和阶段状态

- 指令：把用户消息、系统回复和当前阶段状态渲染到聊天流和状态区，不接入字幕、音频或数字人。
- 产出：文本会话可视化。
- 验证测试：连续提交三条消息，确认聊天顺序、时间顺序和阶段更新正确；刷新页面后历史消息可恢复。
- 通过标准：文本版会话已具备基础演示价值。

### 步骤 14：实现文本会话导出

- 指令：只实现当前会话的 JSON 导出，包含会话信息、消息列表、阶段变更和基础事件。
- 产出：会话导出能力。
- 验证测试：导出后检查文件是否包含完整会话元数据、用户消息、系统回复和时间戳。
- 通过标准：文本主链路的最小可追溯能力建立完成。

### 步骤 15：补齐 trace 标识

- 指令：为文本主链路中的所有事件和消息补充统一追踪标识，用于串联全链路日志。
- 产出：文本链路的追踪字段。
- 验证测试：任选一轮交互，检查数据库和实时事件中是否都能找到相同追踪标识。
- 通过标准：后续任何问题都可以沿着追踪标识回放。

## 5. 第三阶段：语音主链路基础接入

### 步骤 16：接入麦克风权限和录音控制

- 指令：只实现麦克风授权、开始录音、停止录音和录音状态展示，不上传音频。
- 产出：前端录音控制能力。
- 验证测试：分别测试允许授权和拒绝授权两种情况，确认页面不会崩溃，并能给出清晰提示。
- 通过标准：浏览器侧音频采集稳定可控。

### 步骤 17：实现音频分块上传

- 指令：将录音结果按固定时间窗口分块上传到网关，先不接入 ASR，只保存临时媒体索引。
- 产出：音频上传链路和临时存储记录。
- 验证测试：说一段短句后检查是否生成多个音频块记录；停止录音后确认不会再继续上传。
- 通过标准：前端到网关的音频流动已经建立。

### 步骤 18：实现 ASR 独立服务的离线识别基线

- 指令：只让 ASR 服务支持上传一个完整音频文件并返回整句识别结果，不做流式识别；优先使用企业验证集中的音频片段做输入，但必须先经过预处理规范中的重采样和声道转换。
- 产出：离线识别接口和基础结果结构。
- 验证测试：使用企业验证集中的三段音频样本分别测试，确认都能返回文本、时长和置信度等基础信息，并记录预处理前后音频格式差异。
- 通过标准：ASR 服务能独立工作，具备后续接流式的基础。

### 步骤 18A：生成统一 ASR 输入音频

- 指令：只根据 manifest 把企业验证集音频批量转换为 `16kHz`、`mono` 的派生音频，并把输出路径回填到 `audio_path_16k_mono`，不运行识别。
- 产出：统一 ASR 输入音频目录和回填后的 manifest。
- 验证测试：抽查 NoXI 和 RECOLA 各两条记录，确认派生音频存在、格式正确、`audio_path_16k_mono` 可回查到真实文件。
- 通过标准：后续 ASR 批处理与评测都只依赖规范化音频，不再直接消费原始 `44.1kHz` 双声道文件。

### 步骤 18B：批量写回 ASR 初稿

- 指令：使用步骤 18 的离线识别能力批量处理一小批验证集音频，把识别初稿、分段信息、置信度、模型信息和生成时间写回 `val_transcripts_template.jsonl`，但不得直接写入 `final_text`。
- 产出：带 ASR 初稿的转录模板子集。
- 验证测试：检查被处理样本的 `workflow_status` 是否从 `pending_asr` 变为 `draft_ready`，`next_action` 是否变为 `manual_review`，并确认 `draft_text_raw`、`draft_confidence_mean`、`asr_engine` 等字段已填充。
- 通过标准：ASR 初稿回填流程跑通，且不会破坏未处理样本和已复核样本。

### 步骤 18C：建立人工复核推进流程

- 指令：只针对已有 ASR 初稿的记录建立人工复核推进规则，要求复核者能填写标准化文本、审核结论、问题标记和备注，并将记录推进到 `pending_review` 或 `verified`。
- 产出：人工复核队列和状态推进约定。
- 验证测试：任选两条已有初稿的记录，分别模拟“复核中”和“复核完成”两种情况，确认 `review_status`、`review_decision`、`reviewer`、`reviewed_at`、`final_text`、`text_status` 和 `review_history` 变化正确。
- 通过标准：ASR 初稿之后的人工复核不再依赖临时表格或口头同步。

### 步骤 19：把 ASR 最终文本接入主链路

- 指令：只把离线 ASR 的最终识别结果送回编排服务和前端聊天流，不做 partial transcript。
- 产出：语音输入到文本显示的闭环。
- 验证测试：说一句话后，页面能在合理时间内显示最终转写文本，并触发系统 mock 回复。
- 通过标准：语音已成功替代文本输入进入主链路。

### 步骤 20：升级为流式 partial transcript

- 指令：在不改变最终结果结构的前提下，增加局部转写事件，让前端在用户说话过程中逐步显示文本。
- 产出：流式转写事件。
- 验证测试：用户说一段较长句子时，句子未说完之前页面已出现 partial transcript，停止说话后出现 final transcript。
- 通过标准：语音交互的实时感达到可演示水平。

### 步骤 21：加入静音检测、标点和热词

- 指令：在 ASR 流程中增加静音切分、最终标点恢复和领域热词替换，不修改前端逻辑。
- 产出：增强后的 ASR 输出质量。
- 验证测试：对比增强前后的同一条音频，确认断句更合理、标点更完整、领域词识别更稳定。
- 通过标准：ASR 可读性明显提升，适合直接送入对话模块。

### 步骤 22：建立 ASR 评测基线

- 指令：只整理一批基础评测音频和参考文本，优先从企业验证集里选取代表性音频片段并建立人工校对后的参考文本，输出当前版本的识别误差和句级错误率；评测只允许使用 `workflow_status=verified` 且 `locked_for_eval=true` 的样本。
- 产出：ASR 基线评测报告。
- 验证测试：评测结果能重复生成，报告中包含样本量、误差指标、失败样例、音频来源和当前问题总结；确认参考文本不是完全由模型自评生成，且每条样本都能回溯到人工复核记录。
- 通过标准：后续优化有可对比的客观基线。

补充说明：

- 如果企业验证集当前缺少中文可用参考文本，可并行引入公开中文评测集作为独立链路，例如本地 `MAGICDATA dev+test`
- 公开中文评测集必须走独立 transcript 文件，不得污染企业 `val_transcripts_template.jsonl`
- 公开中文评测集的全量参考目录和冻结评测子集都应生成在本地派生目录，并保持 `locked_for_eval` 语义不变
- 一旦公开中文评测集建立完成，应为其提供一个固定回归入口，例如统一的 `verify_asr_regression`，避免后续每次手工拼接多个校验命令

### 步骤 22A：冻结正式评测参考集

- 指令：从已人工复核完成的记录中挑选一批正式评测样本，显式标记为 `locked_for_eval`，后续除发现明显标注错误外不得随意改写。
- 产出：正式评测参考集和锁定规则。
- 验证测试：检查被锁定样本是否都有 `final_text`、`reviewer`、`reviewed_at` 和明确的审核结论；再次执行转录模板生成脚本时，确认这些记录不会退回 `pending_asr` 或丢失人工复核信息。
- 通过标准：ASR 指标的对比基线固定，后续优化可复现、可对照。

## 6. 第四阶段：真实对话服务与状态机

### 步骤 23：对话服务先接 schema 校验

- 指令：在真实 LLM 接入前，先让对话服务严格按共享契约校验输出格式，mock 数据也必须走同样校验。
- 产出：对话结果结构校验能力。
- 验证测试：故意构造缺失字段和非法阶段值，确认服务会拒绝结果并给出明确错误。
- 通过标准：后续接入真实模型时不会因格式漂移影响全链路。

### 步骤 24：接入真实 LLM，但保持输出结构不变

- 指令：只把 mock 回复替换为真实 LLM 生成，不引入多模态、知识库和复杂规则。
- 产出：真实 LLM 驱动的结构化回复。
- 约束：优先使用已经验证可用的 `openai-compatible` 路径，当前仓库基线推荐 `DashScope compatible-mode/v1 + qwen-plus`，但对外仍统一通过 `LLM_*` 环境变量配置。
- 验证测试：使用五组固定文本样例，确认都能返回合法结构，且单轮响应时间处于可接受范围内。
- 通过标准：真实模型已能稳定承担基础问答。

### 步骤 25：实现会话阶段状态机

- 指令：只增加 engage、assess、intervene、reassess、handoff 五个阶段的流转规则，不改 UI 结构。
- 产出：可追踪的会话阶段机。
- 验证测试：用固定脚本模拟一轮完整会话，检查阶段变化是否符合预期顺序，是否存在跳转异常。
- 通过标准：系统不再是自由聊天，而是按陪护流程推进。

### 步骤 26：加入短期上下文记忆

- 指令：只保存最近若干轮原始对话，用于后续 LLM 上下文拼接，不做长期画像。
- 产出：短期记忆存储。
- 验证测试：先提一个事实型信息，隔两轮再追问，确认系统仍能引用该信息。
- 通过标准：多轮对话具备最基本连续性。

### 步骤 27：加入对话摘要

- 指令：每累计一定轮数生成一次简短摘要，用于减小后续上下文长度。
- 产出：阶段性摘要记录。
- 验证测试：连续对话超过三轮后检查摘要是否生成，刷新页面或重连后摘要仍可被使用。
- 通过标准：长对话不会无限膨胀并破坏稳定性。

### 步骤 28：加入高风险规则前置层

- 指令：在对话生成前增加规则判断，对明显高风险输入优先触发安全路径，不把判断完全交给 LLM。
- 产出：前置风险拦截逻辑。
- 验证测试：输入高风险样例，确认结果直接进入 handoff 或等价安全阶段，即使模型回答偏离也不会绕过拦截。
- 通过标准：安全边界具备硬约束。

### 步骤 29：加入超时与格式失败回退

- 指令：当 LLM 超时、输出非法或为空时，必须返回安全基础模板，不允许主链路中断。
- 产出：对话服务故障回退能力。
- 验证测试：主动模拟 LLM 调用失败，确认页面仍能收到合理回复，且系统状态不会卡死。
- 通过标准：真实模型不稳定时系统仍可演示。

## 7. 第五阶段：TTS 与基础数字人

### 步骤 30：实现单声线 TTS

- 指令：只实现将系统回复文本转为单一声线音频并返回音频地址，不接入数字人动作。
- 产出：TTS 基础服务。
- 验证测试：任选三条系统回复，确认都能生成可播放音频，音频时长与文本长度大致匹配。
- 通过标准：文字回复已升级为可听语音。

### 步骤 31：前端接入语音播放和字幕同步

- 指令：只在数字人区域加入音频播放和字幕显示，暂不加入口型和表情。
- 产出：音频播放与字幕联动。
- 验证测试：播放过程中字幕可见，播放开始和结束状态与界面提示一致。
- 通过标准：系统回复具备基本“被说出来”的感知。

### 步骤 32：接入静态数字人基线

- 指令：先只放一个静态 2D 角色形象，并根据播放状态切换“等待”和“说话中”两种状态，不做复杂动画。
- 产出：基础数字人舞台。
- 验证测试：系统回复时角色状态变化明显，播放结束后回到等待状态。
- 通过标准：已经具备最小数字人形态。

### 步骤 33：加入嘴部开合驱动

- 指令：在不引入复杂模型的前提下，根据音频能量或现成嘴型序列驱动嘴部开合。
- 产出：基础口型联动。
- 验证测试：短句、长句各测试一次，确认说话期间嘴部持续变化，静音期间嘴部关闭。
- 通过标准：数字人不再只是静态播放音频。

### 步骤 34：加入第二个数字人

- 指令：新增第二个角色和第二套声线或表现参数，并允许用户切换，不引入复杂人设记忆。
- 产出：双数字人切换能力。
- 验证测试：切换角色后，页面形象、音色或表达节奏至少有两项明显变化。
- 通过标准：满足“至少两个数字人”基础要求。

### 步骤 35：将对话阶段映射到表情预设

- 指令：根据阶段和情绪字段切换若干预设表情或动作状态，不追求细粒度骨骼动画。
- 产出：阶段驱动的表达差异。
- 验证测试：同一句回复在不同阶段触发时，数字人表现明显不同，且高风险阶段不会出现过度轻快表情。
- 通过标准：数字人表达开始承载业务语义。

### 步骤 35A：接入 3D 面部特征离线验证

- 指令：只建立 3D 面部特征读取和离线对齐验证流程，优先使用企业数据中的 `3D_FV_files` 检查数字人驱动服务的输入接口和时序一致性，不做在线渲染改造。
- 产出：数字人驱动离线验证清单。
- 验证测试：随机抽取一条样本，确认 3D 特征长度与对应情绪 CSV 行数对齐，确认驱动服务能够读取该特征并输出合法的驱动结果结构。
- 通过标准：企业 3D 特征已能用于后续数字人驱动评测，而不是停留在“仓库里有文件但流程未接入”。

## 8. 第六阶段：视频输入与多模态基础

### 步骤 36：接入摄像头权限、预览和抽帧上传

- 指令：只实现摄像头授权、预览和低频抽帧上传，不在前端做复杂视觉推理。
- 产出：视频输入基础链路。
- 验证测试：允许授权和拒绝授权两种场景都能稳定运行；有授权时后端可收到帧索引记录。
- 通过标准：视频模态进入系统但不影响已有主链路。

### 步骤 37：实现情绪面板和三路占位结果

- 指令：先在前端加入情绪面板，显示文本、音频、视频三路结果和最终融合结果，初期可以使用 mock 数据，但必须预留显示企业验证集样本编号和数据来源的位置。
- 产出：多模态结果展示容器。
- 验证测试：页面能够同时展示三路子结果、最终状态、置信度、冲突标记占位以及当前样本来源字段。
- 通过标准：后续多模态能力有稳定挂载位置。

### 步骤 38：实现文本情绪基础分析

- 指令：先用规则或轻量分类方式区分中性、焦虑、低落等基础状态，不要求一次做到精细风险分类；企业验证集中的文本输入优先使用步骤 6E 产生的转录文本。
- 产出：文本路基础标签输出。
- 验证测试：准备三组典型文本样例并补充至少两条来自企业验证集转录文本的样例，确认能得到明显不同的标签结果。
- 通过标准：文本路已具备可用的情绪区分能力。

### 步骤 39：实现音频基础特征分析

- 指令：先提取能量、语速、停顿等基础音频特征，不接复杂情绪模型。
- 产出：音频路基础摘要。
- 验证测试：对比快语速和慢语速、强能量和弱能量样例，并额外抽取两条企业验证集音频样本，确认面板上结果有明显区别，且样本来源信息可追溯。
- 通过标准：音频路具备最小可解释特征。

### 步骤 40：实现视频基础状态分析

- 指令：先判断是否检测到人脸、是否存在明显回避或稳定注视等简单视觉状态，不追求复杂表情识别；企业数据中的视频与 3D 特征只用于离线验证，不直接耦合到在线前端推理。
- 产出：视频路基础结果。
- 验证测试：测试“有人脸”和“无人脸”两种情况，并额外抽取一条企业验证集视频样本做离线验证，确认系统不会因检测失败中断主链路。
- 通过标准：视频路能稳定输出最小状态信息。

### 步骤 41：实现第一版融合规则

- 指令：用简单规则或加权方式融合文本、音频、视频结果，输出统一状态、置信度和冲突标志；离线验证时必须基于 manifest 使用同一片段编号的多模态样本。
- 产出：融合结果基础版本。
- 验证测试：构造“文本中性但音频低能量”的冲突样例，并使用一条 manifest 已对齐的企业样本做离线跑通，确认系统能标记冲突并给出原因说明。
- 通过标准：系统已不再只看文本。

### 步骤 42：将冲突标记接入对话追问

- 指令：一旦融合结果出现冲突，要求对话服务优先生成澄清型追问，不直接进入结论性建议。
- 产出：冲突驱动的追问逻辑。
- 验证测试：输入冲突样例后，系统回复应首先询问用户状态，而不是直接给出干预方案；离线检查时要保留该样本的融合证据字段。
- 通过标准：多模态真正影响对话策略。

## 9. 第七阶段：心理知识库与 RAG

### 步骤 43：建立知识卡片数据集

- 指令：先整理小规模高质量知识卡片，覆盖焦虑支持、低落支持、睡眠建议、呼吸训练和求助引导，不追求百科式规模。
- 产出：结构化知识卡片集。
- 验证测试：人工抽查卡片字段是否完整，是否标明适用阶段、风险等级和禁忌情况。
- 通过标准：知识库内容短而准，可直接进入检索流程。

### 步骤 44：实现知识索引和召回

- 指令：只实现基础向量索引和召回，不接入重排模型，不改对话服务。
- 产出：知识检索 API。
- 验证测试：对三类典型查询分别测试，确认召回结果与主题基本相关，且返回知识标识。
- 通过标准：知识库已具备最小可检索能力。

### 步骤 45：将知识检索结果接入对话服务

- 指令：把当前阶段、风险等级和用户输入一起用于检索，再将返回的知识标识和建议短语注入对话上下文。
- 产出：RAG 驱动的回复增强。
- 验证测试：同一条输入在不同风险等级下，回复引用的知识标识不同，且回复内容与召回知识一致。
- 通过标准：对话已不是纯模型自由发挥。

### 步骤 46：为高风险场景增加知识检索护栏

- 指令：当风险达到高等级时，只允许检索 handoff 或安全支持类知识，不允许普通安抚卡片混入。
- 产出：RAG 风险护栏。
- 验证测试：高风险样例输入后，检查知识引用结果，确认不会返回“普通鼓励一下就好”类内容。
- 通过标准：知识库不会破坏安全策略。

## 10. 第八阶段：日志、评测与部署

### 步骤 47：补齐全链路日志

- 指令：将 ASR、多模态、对话、TTS、数字人事件全部纳入统一日志模型，确保每轮交互都可追溯；凡是来自企业验证集的样本，还要额外记录数据集名称、会话编号、角色编号和片段编号。
- 产出：全链路日志记录。
- 验证测试：任选一轮完整语音交互，检查是否能从输入、转写、融合、检索、回复、TTS、数字人依次回放；若该轮使用企业样本，确认日志中能准确还原样本来源。
- 通过标准：任何一次失败都能快速定位到具体模块。

### 步骤 48：实现会话回放

- 指令：基于导出日志实现会话回放能力，支持在不调用实时模型的情况下重现一次已保存会话。
- 产出：Demo 回放模式。
- 验证测试：选取一场已完成会话，切换到回放模式后确认聊天流、字幕和数字人状态能按时间顺序重现。
- 通过标准：比赛现场具备强保底演示路径。

### 步骤 49：建立时延评测

- 指令：只统计各阶段耗时，包括转写、对话生成、TTS、数字人呈现和总耗时，不做优化动作。
- 产出：时延统计报表。
- 验证测试：执行至少五次完整交互，并补充一轮基于企业验证集离线样本的处理时延记录，确认能得到稳定的平均值和分位值结果。
- 通过标准：后续优化有明确量化目标。

### 步骤 50：建立 10 轮稳定性回归测试

- 指令：只编写并执行固定对话脚本，验证连续 10 轮对话中状态机、上下文、日志和页面是否稳定。
- 产出：10 轮回归结果记录。
- 验证测试：检查 10 轮会话中是否出现上下文丢失、阶段乱跳、连接中断或非法输出；同时抽取企业验证集样本做多模态离线回归，确认新改动没有破坏现有数据适配。
- 通过标准：达到赛题连续对话的基础要求。

### 步骤 51：完成核心服务 Docker 化

- 指令：先只容器化前端、网关、编排、数据库、缓存和对象存储，保证文本闭环可在干净环境启动。
- 产出：核心服务部署配置。
- 验证测试：在新环境中启动后，确认文本创建会话、发送消息、收到回复全流程可用。
- 通过标准：系统开始具备可交付性。

### 步骤 52：把模型服务接入部署配置

- 指令：在核心容器稳定后，再依次加入 ASR、TTS、多模态和对话模型服务，避免一次性全量接入。
- 产出：完整服务部署配置。
- 验证测试：执行一轮完整语音会话，确认音频输入、转写、回复、TTS 和数字人展示均可工作。
- 通过标准：完整主链路可通过统一部署方式启动。

### 步骤 53：执行最终验收清单

- 指令：按赛题要求逐项核对双数字人、文本语音交互、摄像头、麦克风、ASR、知识库、三模态融合、10 轮对话、时延和 Docker 交付状态。
- 产出：最终验收清单和未完成项列表。
- 验证测试：每个赛题要求都要有对应的演示证据、日志证据或评测证据，不能只写“已完成”。
- 通过标准：项目当前状态可直接用于答辩准备和冲刺优化。

## 11. 首轮落地后的优化顺序

基础闭环全部通过后，再按以下顺序进入第二轮优化：

1. 优化 ASR 指标，缩小 WER 和 SER。
2. 将多模态融合从规则升级为轻量学习模型。
3. 强化数字人口型、表情和动作细节。
4. 扩充知识库并加入重排模型。
5. 加强高风险识别召回和异常恢复策略。
6. 补齐压测、演示脚本、答辩图表和模型工程整理。

## 12. 对 AI 开发者的交付要求

每一步交付时必须同时提交以下内容：

- 本步修改了什么。
- 本步没有修改什么。
- 本步如何验证通过。
- 本步遗留风险是什么。
- 下一步最合理的唯一任务是什么。

这样可以确保开发过程始终保持“范围受控、可验证、可回退”。
